# 小红书文案智能体 · 设计文档

- 日期:2026-06-15
- 状态:已确认,待进入实现计划
- 参考项目:[langchain-ai/deepagents](https://github.com/langchain-ai/deepagents)(开箱即用的 Agent 框架,建于 LangGraph 之上)

## 一、项目定位

基于 deepagents 搭建一个**通用智能体引擎**,对齐官方 "agent = folder" 范式。
"小红书文案创作"是跑在引擎之上的**应用**,通过 **Skills** 机制挂载,可持续增加新玩法。

```
应用层(Skills,可持续加):
  ▸ 按方向产出选题+文案   ← 第一步只做这个
  ▸ (以后)运营一个账号 / 更多玩法
─────────────────────────────────────────
底层引擎(通用,不绑死任何单一玩法):
  对话 / 规划 / 子智能体 / 文件系统 / 数据接入 / 多会话 / 多用户
```

设计原则:底层通用、上层应用通过 Skill 挂载、改一层不动另一层。

## 二、核心能力与约束

- **研究来源**:不联网搜索。智能体的"研究"= 基于**私有数据库**(飞书多维表格)做分析与沉淀。
- **创作闭环**:读爆款数据 → 拆解套路 → 沉淀风格 → 产出新文案。私域数据驱动的创作闭环。
- **交互**:对话式,流式输出,像 Claude 一样可看到规划/工具调用/子智能体进度。
- **多会话**:像 Claude 一样的会话列表,各会话独立、可切换回看。
- **多用户**:20 人团队,共享爆款库 + 各自会话隔离。

## 三、技术栈

| 层 | 选型 |
|---|---|
| 核心引擎 | deepagents (Python ≥3.11) + LangGraph |
| 模型 | 主智能体默认 Claude;子智能体默认便宜快模型;`deepagents.toml` 可改 |
| 前端 | 官方 Agent Chat UI(Next.js,类 Claude 聊天界面) |
| 会话持久化 | Postgres(本地 Docker;以后迁腾讯云 PostgreSQL) |
| 文件 backend | CompositeBackend(StateBackend 隔离 + StoreBackend 共享) |
| 登录 | 飞书 OAuth(本地阶段先用可切换模拟用户;上云再接真飞书) |
| 数据源 | 飞书多维表格(阶段一即接,后端主动调 API,本地可行) |
| 运行环境 | 本地优先(以后整体迁腾讯云) |

## 四、项目结构(对齐官方 folder 范式)

```
小红书智能体/
├── AGENTS.md                 # 主智能体指令:角色、对话风格、何时调子智能体
├── skills/
│   └── topic-content/SKILL.md   # 【应用①】按方向产出选题+文案
├── subagents.yaml            # 分析子智能体(拆解爆款/对标,独立上下文)
├── tools/
│   ├── feishu_bitable.py     # 飞书多维表格读取工具
│   └── auth.py               # 登录鉴权层(本地模拟用户 / 飞书 OAuth)
├── agent.py                  # create_deep_agent 组装入口
├── langgraph.json            # LangGraph server 配置(graphs + env)
├── deepagents.toml           # 模型/部署配置
├── pyproject.toml            # Python 依赖
├── docker-compose.yml        # 阶段一仅编排 Postgres;交付期再整体容器化
└── .env.example              # API keys、飞书凭证、Postgres 连接串
```

虚拟文件系统(运行时,由 CompositeBackend 按路径前缀路由):
```
/drafts/   → StateBackend   每个会话独立的草稿/文案产出   🔒 隔离
/shared/   → StoreBackend   风格沉淀 xhs-style.md          🌐 共享
飞书数据     → feishu_bitable 工具,直接 API 拉,不走文件系统
```

## 五、应用①工作流:"按方向产出选题+文案"

采用**两步式**交互(先选题菜单,你挑,再写文案):

```
你在会话里说一个方向(如"露营装备")
  1. 调 feishu_bitable 工具读飞书表里相关的爆款/对标内容
  2. 派【分析子智能体】拆解:选题角度、标题套路、结构、情绪点
     (独立上下文,拆完把结论写进文件,不污染主对话;阶段一不做缓存,每次实时拆)
  3. 读 /shared/xhs-style.md 当前沉淀的风格
  4. 产出 3~5 个选题方向(选题菜单)
  5. 你选定(在对话里回"第 2 个") →
  6. 写完整文案(标题 + 正文 + 话题标签),在对话流里以可复制格式分块输出,
     同时存一份到 /drafts/ 备查
  7. 多轮打磨(你提意见,它改)
  8. 把新提炼的套路追加更新进 /shared/xhs-style.md(自动沉淀)
```

**飞书读表方式(做法 B:动态自适应)**:后端把整张表的列名 + 数据行原样返回,智能体自己理解哪列是标题/正文/互动数据,**不写死字段映射**。表结构变化不需改代码。可选地在配置里给字段提示(如"互动数据看‘点赞’‘收藏’列"),智能体优先参考。

**风格沉淀**:`/shared/xhs-style.md` 是一个**活文件**,起步近乎空白(仅骨架),每次分析爆款由智能体追加更新,越用越形成本数据集独有的爆款方法论。人不直接编辑,不分管理员。

## 六、多用户(20 人)· 共享库 + 各自会话

| 资源 | 共享/隔离 | 实现 |
|---|---|---|
| 飞书爆款库 | 🌐 全团队共享(只读) | 飞书表,人/导入放入,智能体只读 |
| 风格 `/shared/xhs-style.md` | 🌐 共享,智能体自动沉淀 | StoreBackend(持久、跨用户) |
| Skills | 🌐 共享 | folder 内 |
| 草稿/文案产出 `/drafts/` | 🔒 各会话独立 | StateBackend(随会话) |
| 会话/对话历史 | 🔒 按用户身份隔离 | LangGraph checkpointing + 用户归属 |

**隔离机制**:CompositeBackend 按路径前缀路由——`/drafts/` 走 StateBackend(天然随会话隔离),`/shared/` 走 StoreBackend(跨会话/跨用户共享)。官方 `deploy-content-writer` 例子即此模式。

**登录**:飞书 OAuth → 拿到飞书用户 ID → 作为会话隔离依据。
- 飞书"登录 OAuth"需公网回调地址,本地 localhost 接不了 → 本地用**可切换的 2~3 个模拟用户**占位(能真测隔离逻辑)。
- 飞书"多维表格读取"是后端主动调 API,**不需公网回调,本地即可用**。
- 上云后,模拟用户接口替换为真飞书 OAuth,接口不变。

## 七、明确不做(范围外,结构预留)

- ❌ 配图(生图 / 配图建议)
- ❌ 联网搜索(只用私有数据库)
- ❌ 自动发布
- ❌ 真飞书登录(本地先用模拟用户)
- ❌ 腾讯云部署(先本地)
- ❌ "运营账号"应用(第二个 skill,以后加)
- ❌ 飞书写回 / 双向同步(阶段一只读)

## 八、分阶段落地

**阶段一(现在,本地)** —— 再拆两步降低风险:
- **1a 最小闭环**:引擎 + 应用① + 飞书读取 + 单会话。
  先证明"读飞书爆款 → 拆解 → 出文案"这条核心价值链通且文案质量可用。
- **1b 多用户化**:加 CompositeBackend(State+Store)+ 多会话 + 模拟多用户隔离 + Postgres + 官方前端。
  先 1a 验证价值,再 1b 加多人;若 1a 发现文案质量不行,不必做 1b。

**阶段二**:接真飞书登录(OAuth)+ 迁腾讯云。
**阶段三**:加"运营账号"应用、配图等。

## 九、已知风险与待验证项

1. **风格文件并发写**:`/shared/xhs-style.md` 全团队共享可写,20 人并发可能冲突。阶段一用"追加不覆盖"降低概率,但 StoreBackend 并发追加是否原子**未经源码验证**,标为待验证;高频并发再考虑加锁/队列。
2. **飞书动态读表的理解偏差**:做法 B 依赖模型理解表结构,列名异常时可能偏(小红书常规字段风险低);用可选字段提示兜底。
3. **阶段一体量**:已包含引擎+应用+飞书+多会话+隔离+Postgres+前端,体量不小;故拆 1a/1b,先验证核心价值。
4. **"信任 LLM"安全模型**:deepagents 默认信任 LLM,边界需在工具/权限层设防(飞书只读、不暴露 secret)。

## 十、需要用户提供的凭证(开搭时)

- 飞书企业自建应用的 `app_id` / `app_secret`(开通多维表格读权限并发布)
- 飞书爆款多维表格的 `app_token` + `table_id`(从表格 URL 获取)
- 模型 API key:`ANTHROPIC_API_KEY`(主)/ `OPENAI_API_KEY`(可选)

> secret 为敏感信息,填入 `.env`,不入库、不回显。
